# /tsh-plan

:::info
Not invoked directly by users. To trigger implementation planning, use [`/tsh-implement`](../public/implement) ‚Äî the [Engineering Manager](../../agents/engineering-manager) will automatically delegate to the [Architect](../../agents/architect) when a plan is needed.
:::

**Agent:** Architect  
**File:** `.github/internal-prompts/tsh-plan.prompt.md`

Creates a detailed, phased implementation plan from the research context.

## How It's Triggered

```text
/tsh-implement <JIRA_ID or task description>
```

The Engineering Manager identifies that no implementation plan exists and delegates planning to the Architect automatically.

## What It Does

1. **Analyzes context** ‚Äî Reviews the `.research.md` file and cross-checks with best practices.
2. **Analyzes tech stack** ‚Äî Identifies domain-specific best practices.
3. **Verifies current implementation** ‚Äî Searches the codebase for existing components, functions, and utilities related to the feature.
4. **Understands project standards** ‚Äî Reviews `*.instructions.md` files.
5. **Prepares implementation plan** ‚Äî Creates a self-contained plan that lower-tier implementors can execute without reopening research for basic context.
6. **Defines core plan sections** ‚Äî Always includes `Glossary / Ubiquitous Language`, `Technical Context`, and `Traps and Warnings`.
7. **Defines phases and tasks** ‚Äî Each phase gets a reusable preamble (`Purpose`, `State Before`, `State After`, `Dependencies / Risks`), and each task has a clear title, action type (`[CREATE]`/`[MODIFY]`/`[REUSE]`), `Context`, `Approach`, `References`, `Traps`, and definition of done checklist.
8. **Addresses security** ‚Äî Includes security considerations.
9. **Defines testing** ‚Äî Guidelines for validation.
10. **Controls scope** ‚Äî Only plans changes for THIS task; documents improvements separately.
11. **Supports validation** ‚Äî The Engineering Manager sends the finished plan to the Architect Reviewer via the internal `/tsh-review-plan` prompt before implementation begins.

## Skills Loaded

- `tsh-architecture-designing` ‚Äî Architecture design process and plan template.
- `tsh-codebase-analysing` ‚Äî Analyze existing codebase.
- `tsh-implementation-gap-analysing` ‚Äî Verify what exists vs what needs to be built.
- `tsh-technical-context-discovering` ‚Äî Understand project conventions and patterns.
- `tsh-sql-and-database-understanding` ‚Äî When the feature involves database changes.

## Output

A `.plan.md` file placed in `specifications/<task-name>/`:

```text
specifications/
  user-authentication/
    user-authentication.research.md
    user-authentication.plan.md    ‚Üê new
    user-authentication.plan-review.md ‚Üê generated by the Architect Reviewer
```

The plan includes a glossary, embedded technical context, traps and warnings, checklist-style phases with reusable preambles, tasks with `[CREATE]`/`[MODIFY]`/`[REUSE]` action types plus `Context` / `Approach` / `References` / `Traps`, acceptance criteria, security considerations, and testing guidelines. Plans remain guidance artifacts only and must not contain real / production code.

:::tip
Review the plan thoroughly. Confirm scope, phases, and acceptance criteria before starting implementation.
:::
